# TriageAI React Frontend v2.1

Premium React + TypeScript frontend for the TriageAI / SympDirect ESI clinical intake and care-routing workflow.

This is structured as a real product frontend, not a throwaway UI prototype. It runs backend-connected by default:

```text
React + TypeScript frontend
→ FastAPI backend on port 8001
→ SQLite for local development / PostgreSQL for Azure deployment
→ final LightGBM model artifacts
→ safety gate escalation
→ clinician review / override
→ audit trail
→ official backend PDF report generation
```

## Backend endpoints used by the React app

```text
GET    /ready
GET    /dashboard/summary
GET    /assessments
GET    /assessments/{assessment_id}
GET    /assessments/{assessment_id}/audit
GET    /assessments/{assessment_id}/report/pdf
POST   /predict
POST   /clinician-review
```

The backend/database is the source of truth for real assessments, dashboard summaries, audit events, clinician review, and official reports. Mock data is retained only for tests and explicit isolated UI development.

## PDF strategy

The official report is generated by the backend ReportLab service through `GET /assessments/{assessment_id}/report/pdf`. The legacy browser-side PDF helper remains only for mock-mode experiments and is not used as the official report source.

## Database and deployment

Local development uses SQLite via `DATABASE_URL=sqlite:///./triageai.db`. Azure or other production deployments should use PostgreSQL. Keep generated local files such as `node_modules`, `dist`, `*.tsbuildinfo`, local SQLite databases, and generated PDFs out of git.

## Screenshot-ready demo data

From the repository root, run:

```bash
python scripts/reset_demo_data.py
```

The command is local SQLite only. It backs up the existing database, clears demo workflow tables, seeds named patients with MRNs, runs predictions through the backend model path, saves clinician review/audit records, and prepares backend PDF report records for the Reports page.

## Safety wording

This project is clinical decision support only. It is not a diagnostic tool and does not replace clinicians. Use wording such as structured intake, ESI care routing, safety-rule escalation, clinician review, human-in-the-loop override, audit trail, and PDF decision-support summary.

## v2.2 intake UI correction

This version adjusts the New Intake screen based on bedside usability feedback:

- The New Assessment page is now a fixed one-screen clinical workstation on desktop.
- The active intake panel, stepper, action buttons, and live summary stay in one viewport.
- The vitals section is the only area designed for horizontal scrolling.
- Vitals sliders now show stable normal reference ranges/bands while the current value changes.
- Age-aware flags remain active, but the normal range text stays visible so the user can compare current value vs reference without losing context.

The goal is closer to a real hospital intake console: less full-page scrolling, faster clinical scanning, and more stable vital-sign interpretation.
